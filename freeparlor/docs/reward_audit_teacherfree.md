## 1. 現行報酬の実体（実装ベース）

`calc_delta_blend` は **per-kyoku デルタ** の系列を返す（GRP feature の局境界ごと）。

```
reward = alpha·sotensu + gamma_pt·juni + beta·chip_deltas·chip_value + opp
```

各項の実装対応：

- `sotensu = calc_delta_points(...) / 1000.0`
  … その局での自分の素点増減（生スコア差）を 1000 で正規化。
- `juni = calc_delta_pt(...)`
  … GRP が予測する着順確率 × ウマ `pts=[3,1,-1,-3]` の期待値（`rank_prob @ pts`）の、局間差分。
    Mortal 本来の GRP ベース報酬シェーピング。
- チップ項 `beta * chip_deltas * chip_value`（`chip_value=5.0`）
  … その局のチップ増減（生デルタ）。
- `opp = -beta * lambda_opp * chip_value * aka_held * w * fire`
  … `fire = (won==0) & (dealt_in==0) & (aka_held>0)`、`w = tenpai_end ? 1.0 : noten_factor`。
    「赤を持っていたが和了れず・放銃もせず局が終わった」場合の取りこぼしペナルティ。

現行の健全パラメータ（`next_steps_2.md §8`）：offline `beta≤0.3, lambda_opp=0.3`、
`pts=[3,1,-1,-3]`（天鳳ウマ）。online はこれ + `enable_cql_online=true`。

## 3. 棚卸しの主要発見

### 発見 A（最重要）: 報酬は副露について一言も言っていない

素点・順位・チップ・opp のどこにも「鳴く / 鳴かない」を**直接**評価する項が無い。
副露は「鳴いた結果和了れば素点/順位/チップが入る」という**間接報酬でしか効いていない**。

帰結：**鳴き頻度の制御は、報酬が一切やっていなかった。100% CQL（教師分布への引き戻し）が代行していた。**

- main（CQL off）＝報酬に鳴き抑制が無い → 副露61%暴走（報酬が止めていないので当然）。
- mqw03（CQL on）＝CQL だけが蓋 → その蓋が step で振動 → 本日の崩落（P(call) 半減）。

**教師を捨てる ＝ 唯一の鳴き頻度制御装置を捨てる。**
`next_steps_2.md §8` の「`enable_cql_online=false` だと無差別鳴き暴走」は、この空白の別表現。

### 発見 B: 着順とチップの価値ストリームが非対称に密度化されている

GRP（juni 項）は素点しか見ておらず、**チップを一切見ていない**。
結果、順位という価値ストリームだけが GRP で密に shaping され、チップは疎な生デルタのまま。
フリー雀荘の商用品質を「チップ込み収支」で測るなら、この密度の非対称は設計で埋める必要がある。
（Phase5 の Q_chip はこの非対称を後付けヘッドで埋めようとして失敗した。）

### 発見 C: GRP は CQL ほど教師依存ではない

着順ダイナミクスは素点＝スコアで決まるため、打ち筋ではなくルール駆動。
ウマ（`pts`）を対象ルールに合わせれば self-play でも概ね機能する見込み。
CQL のように「打ち筋」を教師分布に縛ってはいない。
※ただし GRP の訓練データ源（現状 2009天鳳と推定、要確認）が着順予測にどれだけ効いているかは
　未検証。教師非依存を徹底するなら self-play 記録での GRP 再訓練 or 生着順への置換も選択肢。

## 4. 結論ブロック（教師 + CQL を捨てた後に空く穴）

**Yes/No 判定:**

1. **鳴き頻度制御の空白 — 存在する: Yes（最重要）**
   CQL が唯一担っていた鳴き頻度制御が、教師撤去で丸ごと消える。
   報酬 or 方策目的（例: エントロピー正則化）で新たに担う仕組みの新設が (あ) の最優先要件。
   これを空白のまま PPO / self-play に移行すると、副露61%暴走が確実に再現する。

2. **チップの疎・非対称 — 要再設計: Yes**
   GRP で順位だけ密化、チップは疎な後付け。後付け Q_chip ヘッドは不活性で失敗済み。
   チップを first-class に統合する設計（別ヘッド TD ではなく主報酬 / advantage への統合）が必要。

3. **GRP の扱い — 要判断（依存は軽い）**
   着順ダイナミクスはルール駆動のため流用余地あり。
   判断事項: (i) ウマ `pts` を対象（フリー雀荘）ルールに合わせる、(ii) チップ盲点をどう埋めるか、
   (iii) 教師非依存を徹底するなら self-play で GRP 再訓練 or 生着順へ置換するか。

**維持 / 廃棄:**
- 素点 sotensu: **維持**（ルール非依存・無傷）。
- opp: **廃棄候補**（階層違いのパッチ + 守備歪めリスク）。

**次工程 (あ):** 上記 1 → 2 → 3 の順で報酬を再設計する。
起点は「1. 教師なしで鳴き頻度を正しい水準に収束させる仕組みをどう報酬 / 方策目的に組み込むか」。
